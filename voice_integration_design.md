# 语音识别集成设计方案

## 目标
实现飞书语音消息的自动识别和语音回复功能。

## 当前状态
### ✅ 已完成
1. **Azure语音识别：** 准确识别中文语音
2. **语音处理管道：** 端到端处理流程
3. **Azure TTS：** 高质量语音合成
4. **飞书消息发送：** 支持文字和语音消息

### 🔄 待完成
1. **自动触发机制：** 收到语音消息时自动处理
2. **错误处理：** 识别失败时的降级方案
3. **配置管理：** OpenClaw集成配置

## 集成方案选择

### 方案1：OpenClaw插件扩展
**描述：** 扩展feishu插件，添加语音消息处理钩子
**优点：**
- 原生集成，性能好
- 统一配置管理
- 跟随OpenClaw更新

**挑战：**
- 需要TypeScript/Node.js开发
- 需要理解OpenClaw插件架构
- 开发周期较长

### 方案2：中间件脚本模式
**描述：** 创建独立的Python脚本，监听消息并处理
**优点：**
- 快速原型，使用现有Python脚本
- 与OpenClaw解耦
- 易于调试和维护

**缺点：**
- 需要额外的进程管理
- 消息传递可能延迟
- 配置需要同步

### 方案3：混合模式
**描述：** 使用OpenClaw的cron任务或系统事件触发
**实现：**
1. 配置cron任务定期检查新语音消息
2. 发现新消息后调用处理脚本
3. 发送回复消息

**优点：**
- 利用现有OpenClaw功能
- 无需插件开发
- 简单可靠

## 技术细节

### 文件结构
```
workspace/
├── voice_processing_pipeline.py     # 核心处理管道
├── feishu_voice_handler.py         # 飞书语音处理器
├── azure_speech_recognition.py     # Azure语音识别
├── azure_tts_simple.py             # Azure TTS
└── config/
    └── voice_integration.json      # 集成配置
```

### 处理流程
```
1. 用户发送语音消息 → 飞书
2. OpenClaw接收并保存音频文件
3. Cron任务触发检查脚本
4. 脚本检测到新语音消息
5. 调用语音处理管道：
   a. 语音识别 (Azure)
   b. AI回复生成 (当前模型)
   c. 语音合成 (Azure TTS)
6. 发送回复：
   a. 文字回复 (识别结果 + AI回复)
   b. 语音回复 (合成音频)
```

### 错误处理策略
1. **Azure服务失败：**
   - 重试1次
   - 降级为纯文字回复
   - 记录错误日志

2. **音频格式问题：**
   - 使用ffmpeg转换格式
   - 支持常见格式(ogg, mp3, wav)

3. **网络问题：**
   - 超时设置(10秒)
   - 本地缓存最近结果
